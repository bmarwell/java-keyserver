= Java Keyserver — short architecture snapshot
:toc:
:toclevels: 2

This is a compact ARC42-style snapshot of the repository as it stands today.
It is intentionally brief because the codebase is still evolving.

== 1. Introduction and goals

The project aims to provide a PGP/GnuPG keyserver in Java with separate web
surfaces for the OpenPGP PKS protocol and a REST API. The intended operating
modes are local, syncing, mirror, and group-based lookup.

== 2. Constraints

* Jakarta EE is the main application model: CDI, JAX-RS, JPA, and JTA.
* Open Liberty is the runtime target for the web modules.
* PostgreSQL is used for persistence, with Flyway-managed schema changes.
* Bouncy Castle handles OpenPGP parsing and key handling.

== 3. Solution strategy

The code follows strict hexagonal architecture (ports & adapters).
The application core knows nothing about HTTP, JPA, or SMTP — it only speaks in
terms of ports (interfaces).  Adapters translate between the external world and
those ports.

=== Primary (inbound) ports — the application's public API

Defined in `application-api`.  Adapters outside the hexagon call in through these.

|===

=== Layer rules

* `application-api` contains *only* primary ports (commands, responses,
  exceptions, `CommandService`).  It must never contain driven/outbound ports.
* Driven ports live exclusively in `application-port-*`.
* Command handlers are domain logic in `application-core` — not ports themselves.
* Adapters are allowed to depend on ports but never on each other.
* `common/*` modules contain pure value objects and utilities (no Jakarta EE,
  no Bouncycastle — they must remain deployable anywhere).

== 5. Runtime view

Request flow is intended to look like this:

`HTTP endpoint -> command/query -> application service -> handler -> port -> adapter -> database or mail`

For lookup use cases, the flow is shorter:

`HTTP endpoint -> repository service -> persisted key data`

== 6. Weak spots

* `op=index` and `op=vindex` are not implemented (`LookupEndpoint` returns 501).
  GnuPG key search via name/email therefore fails — only fingerprint/key-ID
  `op=get` works. HKP compliance is partial.
* The HKP `KeyServerExceptionMapper` incorrectly returns `application/json`.
  HKP clients (GnuPG, gpg2) expect `text/plain` for error responses.  The REST
  mapper is correct; the HKP mapper must be fixed separately (#132).
* `VerifyEndpoint` always returns `202 Accepted` regardless of outcome.
  The caller never sees success or failure; there is no polling endpoint (#140).
* Async command dispatch is opaque to the HTTP caller: exceptions in the virtual-thread
  executor are logged but not surfaced.  A future iteration needs at minimum a
  correlation-ID header so errors can be traced.
* The `business_transactions.fingerprint` column is always null — no handler
  populates it yet (#139).
* `web/rest` endpoints return `503 Service Unavailable` on every call.
  The module compiles and deploys but provides no usable functionality.
* `SmtpVerificationNotificationAdapter` is not wired. Only the logging adapter
  exists, so no actual verification email is sent.
* No HTTP-layer rate limiting on `POST /pks/add`. The DoS guard in
  `VerifyUidCommandHandler` defends the verification queue but the add path
  is unguarded.
* **DoS on add**: subkey count is not capped (#136). A key with tens of thousands
  of subkeys is fully parsed by Bouncy Castle and later triggers one
  `stripToVerifiedUids` call per verified UID.  `keytext` size is also uncapped
  (#135).
* **BTX left open**: if `recordCompleted()` itself throws after a successful
  handler, the BTX row stays in `STARTED` state forever (#134).
* Command dispatch is runtime-based (`canHandle`) rather than compile-time safe.
  An unregistered command type silently fails at runtime with a warning log.
* `publishVerifiedUid` has a minor TOCTOU: two concurrent first-publish transactions
  both pass the non-locking existence check, both do a full Bouncy Castle parse.
  The `ON CONFLICT DO NOTHING` INSERT serialises correctly, but one of the two
  parses is wasted.  Acceptable in practice; worth noting for honesty.
* No VKS-compatible API (`/vks/v1/…`) yet.  Implementing the keys.openpgp.org
  surface would allow integration tests to reuse the Hagrid test suite (#138).

== 7. Where the design fits well

* The module split is clear and keeps protocol, use case, and storage concerns apart.
* CQRS/command handling matches the PKS add-vs-lookup shape well.
* Value objects reduce stringly-typed plumbing.
* The ports/adapters boundary should make future storage or mail replacements easy.
* Flyway plus JPA is a sensible fit for repository metadata and queue state.
* The full UID-verification flow (scenarios 1–7) is implemented end-to-end:
  submission, token dispatch, expiry guard, concurrent same-fingerprint and
  same-UID race conditions, and armored-key rewriting with exactly the verified
  UID set.  All seven scenarios have unit tests.
* Concurrent UID publication is correctly serialised via
  `INSERT … ON CONFLICT (fingerprint) DO NOTHING` followed by a
  `PESSIMISTIC_WRITE` row lock — no application-level locking needed.
* Third-party certification stripping is in place: only self-signatures are
  stored, preventing the server from becoming a social-graph database.

== 9. Verification flow

=== Happy path

[source]
----
POST /pks/add?keytext=…
  → AddKeyToVerificationQueueCommandHandler
    → JpaVerificationQueueRepository: insert one queue row per UID
    → VerificationNotificationPort: send email with /pks/verify/{token} link

GET /pks/verify/{token}
  → VerifyUidCommandHandler
    → JpaVerificationQueueRepository: find pending row by token
      [guard] token not found         → TokenInvalidException  (400)
      [guard] token expired           → TokenExpiredException  (400)
      [guard] already verified        → return (idempotent, no re-send)
    → JpaKeyRepository.publishVerifiedUid(fingerprint, uidRaw, uidEmail, armoredKey)
      → existence check (non-locking find)
      → if new: parseMasterKey + strip to one UID + INSERT ON CONFLICT DO NOTHING
      → PESSIMISTIC_WRITE find (serialises concurrent updates)
      → reload all verified UIDs + re-strip armored block + updateKey
      → hasUid guard → add UidEntity if absent
    → JpaVerificationQueueRepository: mark row VERIFIED
----

=== Concurrent first-publish (scenario 7)

Two `VerifyUidCommand` calls for different UIDs on the same fingerprint race
at exactly the same moment:

. Both pass the non-locking existence check (`find == null`).
. Both parse the armored key (one parse is wasted — acceptable overhead).
. Both attempt the native INSERT. PostgreSQL serialises on the fingerprint PK:
   the second blocks until the first commits, then observes the conflict and
   skips the insert.
. Both then do a `PESSIMISTIC_WRITE` find, which serialises sequentially.
. Each transaction reloads the current verified-UID set, adds its own UID,
   and re-strips.  The final committed state contains both UIDs.

No UID is lost and no dirty state is written.

== 8. Identity strategy

=== Why surrogate IDs on workflow tables

The `verification_queue` table needs a primary key that is *not* the key fingerprint,
for two reasons:

* **Cardinality** — a single key can have multiple UIDs, each requiring its own
  verification entry, so the fingerprint is not unique within the queue.
  Re-submissions of the same key also create new entries.
* **Security** — the queue PK doubles as the opaque token embedded in the
  verification email link.  Using the fingerprint here would leak which key is
  under verification to anyone who intercepts the URL.  A surrogate ID keeps
  that information out of the token.

The `audit_log` table uses a surrogate key simply because log rows have no
meaningful natural key; the fingerprint and command type can appear many times.

=== Why TSID and not UUID

All surrogate IDs on workflow tables use
https://github.com/vladmihalcea/hypersistence-tsid[TSID] (Time-Sorted Unique
Identifier) stored as `BIGINT` rather than a `UUID`.

[cols="1,1,1"]
|===
|Property |UUID (v4) |TSID

|Storage
|16 bytes
|8 bytes (BIGINT)

|Index behaviour
|Random — causes B-tree page splits and fragmentation
|Time-sorted — sequential inserts remain clustered

|Comparison
|Byte-array or string comparison
|Native 64-bit integer comparison

|URL footprint
|36 chars (`xxxxxxxx-xxxx-…`)
|~13 chars (base-32 canonical form), or just a decimal long
|===

Because TSID encodes a millisecond timestamp, a node identifier, and a
sequence counter into 64 bits, it is unique enough for this workload, globally
unguessable as a verification token, and far friendlier to the PostgreSQL
B-tree index on the `id` column than a random UUID would be.

== 10. Business transactions

=== Motivation

Every mutating command (add key, disable key, revoke key) is tracked in a
`business_transactions` row before any handler logic runs.  This provides:

* A durable audit log that survives handler failures or JTA rollbacks.
* A foundation for admin dashboards and anomaly detection.
* A correlation ID (`btx_id`) that links the audit log entry to the originating
  command invocation.

=== Schema

----
business_transactions
  id            BIGINT (TSID)   PK     -- node-aware, 42-bit ts | 10-bit node | 12-bit seq
  command_type  TEXT NOT NULL
  fingerprint   TEXT            NULL   -- extracted when available
  caller_ip     TEXT            NULL
  state         TEXT NOT NULL          -- STARTED | COMPLETED | FAILED
  started_at    TIMESTAMPTZ NOT NULL
  completed_at  TIMESTAMPTZ     NULL

audit_log
  btx_id        BIGINT NOT NULL  FK → business_transactions(id)
----

=== TSID node layout

10-bit node field allows 1024 nodes × 4096 IDs/ms.  The node ID is read from
the `TSID_NODE` environment variable (0–1023, default 0).  In a multi-instance
deployment each node **must** be assigned a unique value.

=== Transactional guarantees

|===
|Write |JTA transaction |Survives handler rollback?

|BTX STARTED
|`REQUIRES_NEW`
|✓ always

|Handler work
|`REQUIRED` (joins outer)
|n/a

|BTX COMPLETED / FAILED
|`REQUIRES_NEW`
|✓ always
|===

=== Async execution

Commands are dispatched via Jakarta Concurrency 3.1.  A named
`@ManagedExecutorDefinition(virtual = true)` executor (`KeyServerCommandExecutor`)
runs each command on a dedicated virtual thread.  The CDI request scope is
propagated, making `@RequestScoped BusinessTransactionContext` (which carries the
current `btxId`) available throughout the call stack with no parameter passing.

JAX-RS endpoints call `commandService.handleCommand(command)` synchronously and
return `Response.accepted()` immediately — the async work happens inside the executor
without the HTTP thread waiting for it.

== 11. Privacy

=== IP anonymization

No full IP address is ever stored.  Before any persistence or audit write the
request IP is anonymized:

* **IPv4** — last octet zeroed (`192.168.1.42` → `192.168.1.0`, i.e. `/24`).
* **IPv6** — last 80 bits zeroed, keeping the `/48` prefix
  (`2001:db8:cafe::1` → `2001:db8:cafe::`).

The anonymization happens in `IpAnonymizer` (in `common/ids`) and is applied by
the primary adapter (`AddEndpoint`) *before* building the command.  The command
object therefore always carries an already-anonymized IP; no component
downstream needs to think about this.

=== Third-party certification stripping

All third-party certifications (signatures made by keys other than the uploading
key itself) are stripped before a key is stored in the `keys` table.  This
prevents the keyserver from becoming a social-graph database.  Only self-sigs
are retained (key bindings, UID certifications, subkey bindings).

=== Observations on keys.openpgp.org (Hagrid)

We take inspiration from Hagrid.  Its good ideas:

* UIDs are withheld from search results until the email address is verified.
* Third-party certifications are stripped (privacy, no WoT publishing).
* GDPR key/UID deletion is supported.
* Fingerprint-only access without verification is allowed.

Its weak points that we address differently:

* **Hardcoded token TTL** — ours is configurable per deployment.
* **Email-only verification** — our `VerificationNotificationPort` abstraction
  allows alternative channels (FIDO, phone) without changing the core.
* **No audit trail** — we keep a `business_transactions` log for operators.
* **Metadata leakage** — Hagrid stores keys verbatim after verification;
  we strip third-party sigs before the `keys` row is written.

== 12. Current state

The add → verify → lookup path for `op=get` is functionally complete:
a key can be submitted, its UIDs verified by email link, and retrieved
by fingerprint, key ID, or email.

Remaining gaps: `op=index`/`op=vindex` not implemented, SMTP not wired,
`web/rest` stubs only, no HTTP rate limiting on add.  The architecture is
sound; the functional surface is narrow.

This document is kept deliberately short.  Section numbers reflect deliberate
ordering; section 9 was added to fill the gap that existed between §8 and §10.
