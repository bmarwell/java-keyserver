= Implementation Plan
:toc:
:toclevels: 3

This document captures the planned work to make the keyserver functionally
compliant with the HKP protocol (the protocol GnuPG uses by default).
It covers protocol requirements, database schema, exception handling, transaction
discipline, and the specific changes needed in each module.

== Status snapshot (2026-05)

The original plan has partly been implemented and now needs to be read as a
living document rather than a greenfield checklist.

Already in place on `main`:

* `POST /pks/add` uses `application/x-www-form-urlencoded` with a `keytext`
  parameter.
* `GET /pks/lookup?op=get` works for fingerprint, long/short key ID, and email search.
* `GET /pks/verify/{token}` consumes verification tokens and publishes verified UIDs.
* HKP error responses are `text/plain`.
* The add path enforces a configurable UTF-8 keytext size limit via
  `keyserver.pks.max-key-bytes`.

Still open and reflected by current GitHub issues:

* `op=index` / `op=vindex` are not implemented yet (#137).
* `VerifyEndpoint` still returns `202 Accepted` without user-visible success/failure (#140).
* `business_transactions.fingerprint` is still not populated (#139).
* `recordCompleted()` can still leave BTX rows stuck in `STARTED` if it fails (#134).
* Subkey-count limiting and the remaining add-path config knobs are still pending (#136, #141).
* `web/rest` still needs a real response contract and RFC 7807 error bodies (#133).

== 1. HKP Protocol Requirements

The HTTP Keyserver Protocol (HKP, informally standardized by SKS/GnuPG interoperability
and recently captured in IETF draft-gallagher-openpgp-hkp) requires three operations.

=== 1.1 Lookup (`GET /pks/lookup`)

Required query parameters:

[cols="1,1,3"]
|===
|Parameter |Values |Notes

|`op`
|`get`, `index`, `vindex`
|`get` returns armored key data; `index`/`vindex` return listing

|`search`
|`0x<keyid>`, `0x<fingerprint>`, email, or arbitrary string
|Must accept short (8 hex) and long (16 hex) key IDs, v4/v5 fingerprints,
and email addresses.

|`options`
|`mr` (machine-readable)
|When set, `index`/`vindex` must return the `info:1:N` text format.

|`exact`
|`on`
|When set, no substring matching — only exact fingerprint/key-ID/email matches.
|===

The `op=get` response must be `Content-Type: application/pgp-keys` with
an ASCII-armored public-key block (`BEGIN PGP PUBLIC KEY BLOCK`).

The `op=index` machine-readable format:

----
info:1:<N>
pub:<fingerprint>:<algo>:<keylen>:<ctime>:<exptime>:<flags>
uid:<escaped-uid>:<ctime>:<exptime>:<flags>
----

Flags include `r` (revoked), `d` (disabled), `e` (expired).

=== 1.2 Add (`POST /pks/add`)

The request body is `application/x-www-form-urlencoded` with a single field:

----
keytext=<url-encoded ASCII-armored public key block>
----

NOTE: This is implemented on `main`: `AddEndpoint` accepts a form parameter
named `keytext`.  The add path also rejects oversized submissions using the
configurable `keyserver.pks.max-key-bytes` limit.

The success response is HTTP 200 with a plain-text confirmation body.
On failure, HTTP 4xx with a plain-text error message.

=== 1.3 Commands Needed

[cols="1,2"]
|===
|Command |Trigger

|`AddKeyToVerificationQueueCommand`
|`POST /pks/add` — implemented; remaining follow-up is add-path hardening/configuration.

|`VerifyUidCommand`
|Email link click — verifies one UID and promotes key to published state.

|`LookupKeyByIdCommand` (or query)
|`GET /pks/lookup?op=get` — lookup by key ID or fingerprint.

|`IndexKeyCommand` (or query)
|`GET /pks/lookup?op=index` — listing format.

|`RevokeKeyCommand`
|Future: via signed revocation certificate upload.
|===

NOTE: `Lookup` and `Index` are read-only queries without side effects.  Consider
introducing a symmetric `QueryService` / `KeyServerQuery` hierarchy so they do
not go through the command path and require no `@Transactional` write.

TIP: The most valuable next protocol step is still `op=index` (#137) because it
unblocks `gpg --search-keys` and turns the existing lookup support into a more
useful HKP surface.

== 2. Fields to Extract and Store

=== 2.1 Keys Table (expand existing `keys`)

The existing `keys` table only has `rfingerprint`, `ctime`, `mtime`, `md5`.
It must be expanded to hold everything needed for lookup responses and
machine-readable output.

[cols="1,1,3"]
|===
|Column |Type |Notes

|`fingerprint`
|`TEXT NOT NULL PRIMARY KEY`
|Full hex fingerprint (40 chars v4, 64 chars v5). Replace `rfingerprint` as
primary key; keep `rfingerprint` as indexed column for suffix-search.

|`rfingerprint`
|`TEXT NOT NULL`
|Reversed fingerprint for suffix search (kept for compatibility with SKS
indexing style).

|`keyid_long`
|`TEXT NOT NULL`
|Last 16 hex chars of fingerprint.

|`version`
|`INTEGER NOT NULL`
|OpenPGP key version: 4 or 5.

|`algorithm`
|`INTEGER NOT NULL`
|OpenPGP algorithm tag (e.g., 22 = Ed25519, 1 = RSA).

|`bit_strength`
|`INTEGER`
|Bit length for RSA/DSA; null for fixed-length ECC.

|`creation_time`
|`TIMESTAMP WITH TIME ZONE NOT NULL`
|From the key packet.

|`expiration_time`
|`TIMESTAMP WITH TIME ZONE`
|Null if the key does not expire.

|`revoked`
|`BOOLEAN NOT NULL DEFAULT FALSE`
|Set when a revocation certificate is processed.

|`armored_key`
|`TEXT NOT NULL`
|Full ASCII-armored public key block for `op=get` responses.

|`md5`
|`TEXT NOT NULL UNIQUE`
|MD5 of the armored key (already present — retain for compatibility).

|`ctime`
|`TIMESTAMP WITH TIME ZONE NOT NULL`
|Row creation time (server-side).

|`mtime`
|`TIMESTAMP WITH TIME ZONE NOT NULL`
|Row last-modified time (server-side).
|===

== 3. Exception Hierarchy

All domain exceptions should live in `application/application-api`.
They extend `RuntimeException` so they cross CDI method boundaries without
checked-exception declarations.

=== 3.1 Base Class

[source,java]
----
// package io.github.bmarwell.keyserver.application.api.ex;
public abstract class KeyServerException extends RuntimeException {

    private final Optional<KeyFingerprint> fingerprint;
    private final String correlationId;  // UUID, set by CommandService

    protected KeyServerException(String message, Optional<KeyFingerprint> fingerprint,
                                  String correlationId) { ... }
    protected KeyServerException(String message, Throwable cause,
                                  Optional<KeyFingerprint> fingerprint,
                                  String correlationId) { ... }

    public Optional<KeyFingerprint> getFingerprint() { ... }
    public String getCorrelationId() { ... }
}
----

=== 3.2 Hierarchy

----
KeyServerException  (abstract, application-api)
├── KeyParsingException          — armored input could not be decoded
├── KeyValidationException       — key is syntactically valid but rejected
│   ├── KeyExpiredException      — primary key or all UIDs expired
│   ├── KeyRevokedException      — primary key carries a revocation signature
│   ├── NoVerifiableUidException — no UID has an email address
│   └── InvalidAlgorithmException — algorithm not accepted by this server
├── KeyNotFoundException         — no key matching the search criteria
├── DuplicateKeyException        — fingerprint already published (idempotent add
│                                  should return 200, not throw — design choice)
└── VerificationException        — problems with the email-verify flow
    ├── TokenExpiredException
    └── TokenInvalidException
----

NOTE: `DuplicateKeyException` may be handled silently (200 OK, re-send verification
mail) rather than thrown to the endpoint.  Decide at implementation time; plan
for the exception regardless.

=== 3.3 Placement

* `KeyServerException` and all subclasses → `application/application-api`
  (`io.github.bmarwell.keyserver.application.api.ex`)
* `InvalidKeyIdException` in `common/ids` stays there — it is a value-object
  parsing error, not a business error.

== 4. Exception Mappers

JAX-RS `ExceptionMapper` implementations belong in the web modules
(`web/openpgp-keyserver-protocol` and/or `web/rest`).

[cols="1,1,2"]
|===
|Exception |HTTP Status |Body

|`KeyNotFoundException`
|404 Not Found
|Plain-text `"No key found for <search>"`

|`KeyParsingException`
|400 Bad Request
|Plain-text error message

|`KeyValidationException` and subtypes
|400 Bad Request
|Plain-text error message (include reason: expired, revoked, etc.)

|`DuplicateKeyException`
|200 OK
|Plain-text confirmation (idempotent add)

|`VerificationException` subtypes
|400 Bad Request
|Plain-text reason

|`KeyServerException` (catch-all)
|500 Internal Server Error
|Generic plain-text; log full details server-side only
|===

Each mapper must:

1. Log the exception at the appropriate level (WARN for 4xx, ERROR for 5xx).
2. Include the `correlationId` from the exception in the response header
   (`X-Correlation-ID`) so callers can reference it in support requests.
3. Never expose stack traces in the response body.

== 5. Transaction Strategy

=== 5.1 Current State

`KeyServerCommandService` dispatches commands via an async virtual-thread executor.
The actual handler invocation is delegated to `TransactionalCommandDispatcher` — a
separate `@ApplicationScoped` CDI bean — so the `@Transactional` annotation is
applied through the CDI proxy rather than bypassed by same-bean self-invocation.
Handler DAO calls therefore run in a single JTA `REQUIRED` transaction.  BTX
lifecycle writes (`recordStarted`, `recordCompleted`, `recordFailed`) use
`REQUIRES_NEW` so they survive a handler rollback.

=== 5.2 Target State

The current implementation already achieves the target:

* `dispatch()` lives in `TransactionalCommandDispatcher` (separate bean), so
  `@Transactional` fires correctly through the CDI proxy.
* Each BTX write uses `REQUIRES_NEW`.
* ✅ The anonymized caller IP is forwarded to `recordStarted()` via `CommandCallerContext`
  (see §8.7).
* `business_transactions.fingerprint` is still not populated; track this as #139.

=== 5.3 Audit Log Integration

BTX rows serve as the audit log via the `BusinessTransactionRepository` port.
`KeyServerCommandService` wraps every handler call in the following pattern:

[source,java]
----
btxRepository.recordStarted(btxId, commandType, callerContext.anonymizedCallerIp());
btxContext.initialize(btxId);
try {
    dispatcher.dispatch(command, callerContext);  // @Transactional in separate bean
    btxRepository.recordCompleted(btxId);
} catch (Exception ex) {
    btxRepository.recordFailed(btxId, ex.getClass().getSimpleName(), ex.getMessage());
}
----

`BusinessTransactionRepository` (port in `application-ports`) is implemented by
`PersistentBusinessTransactionRepository` in `repository`.  Every write uses
`REQUIRES_NEW` so it survives a rollback.

=== 6.8 Future enhancement: command verification rollout

The add command now uses **both** of the patterns discussed during #136:

* a **template-method hook** in `AbstractKeyServerCommandHandler`, and
* a **verification registry** composed of small verification steps.

This should be the default direction for future command validation:

* handlers should not inline large validation blocks in `doExecute(...)`
* command-specific validation should live behind a typed
  `CommandVerificationRegistry<T, V>`
* reusable checks should be implemented as small `VerificationStep<I, O>` classes

Future work can move more handlers (for example token verification or future REST
write commands) onto the same structure as soon as they have enough pre-execution
checks to justify it.

== 7. Open Questions / Design Decisions

* *Idempotent add*: Should re-submitting an already-published fingerprint be
  200 OK (re-send verification) or 409 Conflict?  Lean toward 200 + re-send to
  align with key server user expectations.
* *UID filtering*: PKS protocol historically publishes all UIDs.  The
  Verifying Keyserver (VKS) model (keys.openpgp.org) only publishes verified
  UIDs.  Choose one and document it.
* *Algorithm allow-list*: Reject MD5-based v3 keys and DSA < 2048?  Plan for
  a configurable allow-list.
* *Synchronization*: Multi-keyserver sync is out of scope for now.  The
  `RepositoryName` value object is in place but the sync protocol is not planned.

== 8. Business Transactions (BTX)

=== 8.1 Motivation

Every write command (add key, verify UID, revoke key) must be traceable end-to-end
so that administrators can answer questions like:

* "Which commands ran against fingerprint X in the last 24 hours?"
* "Did the add-key command for key Y complete, or did it fail mid-way?"
* "Show me all commands initiated from IP address Z."

A *business transaction* (BTX) is the outer unit of administrative tracking.
It is distinct from the JTA database transaction: a BTX spans one command invocation
from entry to final state, survives JTA rollbacks (it writes its own status in a
separate JTA transaction), and is stored permanently in the `business_transactions`
table for audit purposes.

=== 8.2 Business Transaction Table

[source,sql]
----
CREATE TABLE business_transactions (
    id              BIGINT                   NOT NULL PRIMARY KEY,  -- TSID (app-assigned)
    command_type    TEXT                     NOT NULL,
    fingerprint     TEXT,                    -- NULL if not extractable before dispatch
    caller_ip       TEXT,
    state           TEXT                     NOT NULL,
    started_at      TIMESTAMPTZ              NOT NULL DEFAULT now(),
    completed_at    TIMESTAMPTZ
);
-- state IN ('STARTED', 'COMPLETED', 'FAILED')
----

The `audit_log` table gains a non-nullable foreign key `btx_id BIGINT REFERENCES business_transactions(id)`.

=== 8.3 BTX ID Strategy

BTX IDs use the same TSID approach as `verification_queue`, but via a *node-aware*
`TsidFactory` rather than `TSID.fast()`.

The TSID bit layout with a 10-bit node field:

[cols="1,1,2"]
|===
|Bits |Content |Notes

|42
|Millisecond timestamp
|~139 years before overflow

|10
|Node ID
|0–1023; configured via env var `TSID_NODE` (defaults to 0)

|12
|Per-millisecond sequence counter
|4096 unique IDs per node per millisecond
|===

This supports up to 1024 concurrent server instances without coordination, with
4 096 × 1 024 = ~4 M unique IDs per millisecond globally.

A CDI `@ApplicationScoped` producer creates the `TsidFactory` once at startup,
reading `TSID_NODE` from the environment (or falling back to 0).  All components
that need a TSID inject the factory rather than calling `TSID.fast()`.

=== 8.4 Asynchronous Command Execution

`KeyServerCommandService.handleCommand()` is annotated with
`@ManagedExecutorDefinition` (Jakarta Concurrency 3.1) to define a dedicated
virtual-thread executor, and with `@Asynchronous(executor = "...")` so every
command runs off the HTTP request thread.

[source,java]
----
@ManagedExecutorDefinition(
    name     = "java:app/concurrent/KeyServerCommandExecutor",
    virtual  = true,   // Java 21+ virtual threads
    maxAsync = -1      // no limit; virtual threads are cheap
)
@ApplicationScoped
public class KeyServerCommandService implements CommandService {

    @Asynchronous(executor = "java:app/concurrent/KeyServerCommandExecutor")
    @Override
    public <T extends KeyServerCommand> void handleCommand(T command) { … }
}
----

Benefits of virtual threads here:

* Command handlers perform I/O (PostgreSQL, SMTP).  Virtual threads park during
  I/O without blocking a platform thread.
* The thread pool does not need tuning; the JVM scheduler manages virtual thread
  concurrency automatically.

`CommandService.handleCommand()` returns `void` — it is fire-and-forget from the
caller's perspective.  Exceptions are caught inside the async body and persisted
to the BTX row (`error_type`, `error_message`).  JAX-RS resource methods return
plain `Response` (synchronous); they call `handleCommand()` then return
`202 Accepted` immediately without any `CompletableFuture` or `CompletionStage`
in their signatures.

=== 8.5 BTX Lifecycle Within handleCommand()

[source]
----
handleCommand(cmd):                              (runs in virtual thread)
  1. generate btxId  (TSID via TsidFactory)
  2. btxRepo.recordStarted(btxId, cmd)          REQUIRES_NEW → commits immediately
  3. btxContext.set(btxId)                       @RequestScoped CDI bean
  4. @Transactional: handler.execute(cmd)
  5a. success → btxRepo.recordCompleted(btxId)  REQUIRES_NEW
  5b. failure → btxRepo.recordFailed(btxId, errorType, errorMessage)  REQUIRES_NEW
----

Using `REQUIRES_NEW` for all BTX writes ensures they survive a JTA rollback on the
command's main transaction.

Jakarta Concurrency 3.1 propagates the CDI request context to the async thread,
so the `@RequestScoped` `BusinessTransactionContext` bean is available throughout
the handler's stack without any explicit parameter threading.

=== 8.7 Client IP and Audit Trail

The client IP address is captured and anonymized in the primary adapter
(`AddEndpoint`) before the command is constructed.  The anonymized value is
carried in a `CommandCallerContext` that travels alongside the command — it is
**not** a field on the command record itself.

`AddEndpoint` calls `IpAnonymizer.anonymize(remoteAddr)`, wraps the result in
`CommandCallerContext.of(anonymizedIp)`, and passes both the command and the
context to `CommandService.handleCommand(command, callerContext)`.
`KeyServerCommandService` extracts `callerContext.anonymizedCallerIp()` and
passes it to `BusinessTransactionRepository.recordStarted()`, wiring the
anonymized IP into the BTX audit row.  The context is also forwarded through
`TransactionalCommandDispatcher` to each handler's `execute()` method in case a
handler needs it in the future.

Rules:

* IPv4: last octet zeroed (`192.168.1.42` → `192.168.1.0`)
* IPv6: last 80 bits zeroed (keep `/48` prefix)
* `HttpServletRequest.getRemoteAddr()` is used — reverse-proxy `Forwarded` /
  `X-Forwarded-For` headers are **not** trusted to prevent IP spoofing via forged
  headers.  Configure the proxy to strip and re-add those headers at the network
  edge if needed.

== 9. Verification Notification (First Iteration: Log Only)

The first iteration uses `LoggingVerificationNotificationAdapter` to deliver the
verification URI.  This deliberately logs the full URI (which is a bearer token)
at `INFO` level so that developers and test operators can retrieve the link
without an SMTP server.

**Security note**: The full token MUST NOT appear in production logs once SMTP
is wired.  Either:

* Replace this adapter with `SmtpVerificationNotificationAdapter`, or
* Downgrade the log level to `FINE`/`DEBUG` behind a guarded condition.

The `TODO` comment in `LoggingVerificationNotificationAdapter` marks the location.
See also `.github/AGENTS.adoc § Privacy rules`.

== 10. Self-critique and known design debts

=== 10.1 Things that were done well

* The port/adapter split held cleanly through the VerifyUID work — no leakage
  of JPA types into the core or HTTP types into the repository.
* Concurrent first-publish (scenario 7) is solved with a single `INSERT … ON CONFLICT`
  serialisation point, avoiding application-level locking.
* Indexed short key-ID search uses `rfingerprint` with `text_pattern_ops` —
  `LOWER()` on the column was correctly removed after being introduced initially.
* Verification token values are not logged in exception messages or domain-error
  paths — discovered and fixed during review. The current first-iteration
  `LoggingVerificationNotificationAdapter` remains an explicit, temporary logging
  exception as documented above.

=== 10.2 Honest design debts introduced during implementation

==== TOCTOU in `publishVerifiedUid`

The "skip BC parse if key exists" optimisation uses a non-locking `find` before
the INSERT.  Two concurrent first-publish transactions can both pass that check,
both do a full Bouncy Castle parse, and one of the two parses is discarded.
The `ON CONFLICT DO NOTHING` INSERT still serialises correctly; this is a wasted
CPU cycle, not a correctness bug.

==== `Assertions.fail()` instead of AssertJ in scenario 7 test

`VerifyUidCommandHandlerTest` uses `org.junit.jupiter.api.Assertions.fail()` in
the timeout branch while the rest of the file uses AssertJ.  Should be replaced
with `assertThat(completed).as("threads completed within timeout").isTrue()`.

==== `VerifyEndpoint` always returns 202

The verification endpoint cannot distinguish success from failure in its current
form — the command is dispatched asynchronously and the HTTP thread returns before
the outcome is known.  The user receives no feedback.  A synchronous verification
path (or a status-polling endpoint) is needed for a production-grade UX.

==== `business_transactions.fingerprint` always null

No handler populates the fingerprint column on the BTX row.
Operators cannot filter the audit log by key fingerprint.

==== Error response content type

`LookupEndpoint` returns `text/plain` for 400/404/501 errors.
This is the correct behavior for HKP compatibility: HKP clients expect plain-text
error bodies, so HKP endpoints should continue to produce `text/plain` rather than
`application/problem+json`.  The HKP side is in the right state now.
The deferred work is on the REST side: introduce RFC 7807 responses there (#133)
without regressing HKP compatibility.
